CopyBeat Game



Gameplay Video
Screenshots

Description:

The game has 5 pre-loaded songs packaged in the FMOD master bank. These songs can be selected by pressing on their thumbnail in the song select screen. This prompts you with a beatmap selection box which dynamically loads any saved beatmap files.

After the selection and countdown, buttons play an animation based on the saved trigger value for that button. The time at which the user hits the button as it's flashing determines the type of hit (great/cool/miss). Hits are counted for end of song display. The animation values are found from a beatmap struct that contains a 2d float array for the hit timings. Every frame the 2d float array is iterated through and determines the current button state and its resulting visual state through accompanying variables. Each button is connected to a function that matches the hit button and checks if the button is within a period/has been hit before to determine if any visual change occurs or if the input is ignored.

Beatmaps can be created by the user by pressing 'input mode' (red if active) and selecting the song to make a beatmap for. The user gets a button grid identical to play mode and as the music plays can enter their own periods for animations. Upon hit the button turns yellow and blocks any further input until the period length is over. Many systems are recycled from the play mode for this functionality. At the end of the input, the user can choose to redo/save/play their new map. Save functionality is done using UE4's standard save slots.

Notes:

- The device used in the gameplay video is a chinese tablet dual core processor @1.6GHz and an ARM Mali 400MP (quad core), android 4.1, 1024x768 resolution.
- This game is not complete, I aim to add dynamic user end music loading and beatmap loading through files in the near future as well as overall polish.

RGBA PNG to Pixel Collision Map



GitHub Repository Link
  • This version takes in filename for target image and uses a custom exception class to indicate errors.
  • As the purpose of this program was just to get alpha collision matrices, only RGBA png's will be excepted, otherwise an error is thrown.
  • Input images for alpha output must be square in dimension.

Description:

The reason this program came to be was because I was reading up on retro Sonic guides that went into detail on how slopes worked in Sonic the Hedgehog 1-3. Slopes were pixel perfect and used outcoming rays from the Sonic avatar to determine if there was a block ahead, it it was below the ray height, Sonic would move onto it and otherwise push against it. For more into that, I suggest a read at: http://info.Sonicretro.org/Sonic_Physics_Guide
The idea was such that if a level was being created, the foreground on which Sonic could appear to move on could be used as a collision layer also generated from the pixels. This may be a very expensive way of handling things in terms of collision systems even at the low resolutions the original Sonic games "ran" (:P) at. I haven't tested it as a collision system so I can't say how costly it is.

The program uses libpng to read the file header and decompress the pixel data, then copies the pixel data into heap allocated memory for each row of pixels (each pixel is 4x unsigned char for RGBA). A new text file is created based on the input image file name and iterates through each 4th value in the 2D array of pixel data to get just the alpha value for that pixel and sets the text value for that pixel to 0 if equals 0 and 1 otherwise. The idea behind this being that Sonic can collide with the layer as long as its not entirely transparent. (In terms of adding colour to the background / foreground, think layered images and just using this layer for the alpha-collisions).

The program also offers a functionality for splitting tilemaps into the square dimension images based on the input dimension per image. I've also had it take only RGBA images, but this was more of a tacked on functionality that would add to the main function of the program.

Additional Game-Engine Related Code



GitHub Repository Link
  • Controller code and game loop code make use of SFML graphics for text rendering to screen and window rendering, can be removed if not planning on using those SFML features.

Description:
  1. Game loop uses gameprogrammingpatterns.com game loop style of variable rendering, fixed update where if the visible framerate lags behind, update steps in that loop iteration are increased to matches the average update-per-frame ratio (e.g. 100 physic steps per 60 renders), set to render using SFML graphics, can be replaced. Also set a fixed framerate of 60fps (did this by consuming time in the frame if its < 1/60).
  2. Controller code uses SFML's sf:Joystick class to check controllers' buttons and axis values. The problem with SFML's implementation is that it is reliant on the Windows Multimedia Library (preceeded by DirectInput) and doesn't include out of the box vibration control or separate axis binding of the left and right triggers (on a default xbox360 wired the L2 and R2 are on the same axis as negative and positive).
  3. Collision code assumes that the screen is of a certain resolution and that the collision world is rebuilt every step. For each AABB that we chec collisions for we check what cells of the grid it occupies and add to the list of occupied entities in that cell to the cell data, when the cell data is sorted afterwards, if we get two or more entities in the same cell we then pairwise sweep AABB check them. One known negative of this collision system however is that if multiple collisions happen in a step and a collision causes the end position to be inside another object then it won't be resolved properly.

Demonstration Video
Screenshots

GitHub Repository Link
Description:

This project involved using basic OpenGL 1 and C++ to create one of several tasks. I chose the 3D maze becuase it I believed it would be sufficiently challenging and possibly extensible to another project at some point with its ability to have a top down map and a first person perspective. Although I haven't made any moves thus far in that direction. It would probabaly be best to learn modern OpenGL instead for that if I do end up taking that route.
The program has hard coded walls and includes prevention of movement by the user into a wall if a movement moves into a wall. The solution line is also hard coded but involves using a vector as a stack-like structure where if the camera moves towards the line verticies it pops, and otherwise pushes the previous camera position into the stack.

The turning 'animation' was done by utilising the Idle functionality where I set a flag to cue a custom Idle function that starts a turn and turns off the idle flag when two 45 degree turns are made. Camera turns are based on multiplying the camera by a rotation matrix. This project was completed in four days.

FIRST and FOLLOW set computator



GitHub Repository Link
  • The file is hardcoded to take in the filenames "terminals_input.txt", "language_input.txt" and outputs to "FnF_Sets_Output.txt" (as well as to console).
  • Included in the GitHub Repository Link is the input text files used for Lua 5.1 (output also), any subsititute language can be computed in the same format.

Description:

This program finds the FIRST, FOLLOW and FIRST+ sets that define what elements can be first encountered when reading an expression and what elements can follow it. FIRST+ is an extension of FOLLOW sets where it takes into account whether or not an element that can follow that symbol can be reduced to nothing and transitively that element's FOLLOW set is now set union with the current symbol's FOLLOW set. The operation of this program may take awhile due to the way I have programmed the defining of undefined elements in expressions (to be explained in more detail below). Note: Terminal symbols are literals and do not expand. Non-terminal symbols are symbols that expand to expressions of other terminals or non-terminals.

The program operates in several stages:

  1. Parse the input text files. - This defines an symbol and its productions, some production symbols may not be defined at the time of encounter.
  2. Update all production symbols with defintions.
  3. Compute FIRST sets - this is done recursively.
  4. Compute FOLLOW sets - this is done via looping through each symbol until a definition is found for a symbol, breaks if no update was done after two iterations.
  5. Compute FIRST+ sets - At this point we assume all symbols are defined and apply the FIRST+ rule.

Using looping to find the FOLLOW sets is much less efficient than using recursive, when I first wrote it I didn't put much thought into it. I had some concerns that doing it recusrsively may cause problems with cyclic defintions. While these are on point, and looping would break out if there were no changes and even a helpful message to say "Hey, your grammar has cyclic defintions", but ultimately when defining a grammar cycles should be removed anyways so this concern was unnecessary. I could have changed it but ultimately I was more focused on correct results than speed so I never changed it.
